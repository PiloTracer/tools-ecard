## 1. Executive Summary

E-Cards is a digital card design and batch generation platform. The project is approximately 80% complete: the template designer, batch upload/parsing, storage, auth, and font management are shipped. Remaining work focuses on completing the render worker (card generation pipeline), wiring the batch import mapping service, and hardening existing features.

The plan defines **3 milestones**: M1 replaces mock rendering with real Canvas/Sharp logic, M2 finishes batch workflow (import mapping + records edit), and M3 covers hardening (testing, observability, CI/CD).

**Top risks:** Render worker rendering is mocked (infra exists); batch import returns placeholder responses (routes exist); no CI/CD pipeline.

## 9. Architecture Overview

See foundation doc 04 (`.work/plans/foundation/20260427-04-foundation-architecture.md`).

Three-service architecture:
- **front-cards** (Next.js 16) — UI + BFF API routes
- **api-server** (Fastify + Prisma) — REST + WebSocket + job queue
- **render-worker** (Node.js worker) — BullMQ consumer for card rendering (mock — rendering logic not implemented)

Databases: PostgreSQL 16 (relational), Cassandra 5 (event log), Redis 7 (queue).

## 22. Rollback and Recovery Considerations

- **Database:** Prisma migrations reversible; Cassandra CQL scripts idempotent
- **Render worker:** New rendering runs in separate BullMQ queue; old mock preserved during transition
- **S3 storage:** Generated cards stored with unique keys; no destructive overwrites
