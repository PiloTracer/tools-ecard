## 1. Purpose

Allow operators to **deploy or enter Demo mode** so users can design cards and exercise project/template/font/batch UX **without saving any data on the server**. All durable client state uses browser storage (`localStorage` + IndexedDB). Critical for sales demos and sandboxes on shared infrastructure.

## 2. In scope / Out of scope

**In scope**
- Demo activation via env (`DEMO_MODE` / `NEXT_PUBLIC_DEMO_MODE`) and `/demo` (`?demo=1`)
- Banner, Clear Demo Data, Exit Demo
- Browser persistence for projects, templates/resources, fonts, batches + records
- Synthetic auth (skip OAuth)
- api-server write guard when `DEMO_MODE=true`
- Seed fixtures on first Demo visit (optional defaults)

**Out of scope**
- Full client-side Sharp/Canvas batch render parity with render-worker
- Migrating server data into Demo storage
- Including SeaweedFS in `start.sh` tar.gz backups
- UI Design OS greenfield

## 3. Domain language

| Term | Meaning |
|------|---------|
| Demo mode | Runtime mode where persistence is browser-only |
| Demo repository | Client adapter that replaces API write/read for a domain |
| Demo user | Synthetic authenticated identity (`demo-user`) |
| Write guard | Server middleware rejecting mutating HTTP methods |

## 4. Behavioural spec

- **R1** When `NEXT_PUBLIC_DEMO_MODE` is truthy OR `localStorage['ecards:demo:enabled']==='1'`, `isDemoMode()` is true.
- **R2** Visiting `/demo` or `?demo=1` sets the localStorage flag and redirects to `/dashboard`.
- **R3** While Demo is active, a persistent banner states that data stays in this browser; actions: Clear Demo Data, Exit Demo.
- **R4** While Demo is active, project/template/font/batch service methods that would hit mutating APIs use Demo repositories instead — **no** POST/PUT/PATCH/DELETE to api-server for those domains.
- **R5** Auth: Demo user is authenticated without OAuth; ProtectedRoute allows access.
- **R6** When api-server `DEMO_MODE` is truthy, any mutating method under `/api` (except health) returns **403** with code `demo_mode_readonly`.
- **R7** Clear Demo Data removes all `ecards:demo:*` / demo IndexedDB stores and re-seeds defaults if configured.
- **R8** Exit Demo clears the enabled flag (env-forced Demo cannot fully exit until env is off).
- **R9** Batch “render” in Demo reports mocked/completed client preview status — does not enqueue BullMQ jobs.

## 5. Data model

| Store | Key / objectStore | Contents |
|-------|-------------------|----------|
| localStorage | `ecards:demo:enabled` | `'1'` \| absent |
| localStorage | `ecards:demo:projects` | JSON `Project[]` |
| localStorage | `ecards:demo:selectedProjectId` | string |
| localStorage | `ecards:demo:templates` | JSON template metadata + data refs |
| localStorage | `ecards:demo:batches` | JSON batch list |
| localStorage | `ecards:demo:batchRecords:{id}` | JSON records |
| IndexedDB `ecards-demo` v1 | `blobs` | fonts, images, file payloads by id |
| IndexedDB (existing textile) | resources/templates | reused where already present |

## 6. APIs

No new public HTTP APIs for Demo persistence. Server only adds write-guard behaviour when `DEMO_MODE=true`.

Client: `isDemoMode()`, `enterDemoMode()`, `exitDemoMode()`, `clearDemoData()`.

## 7. Invariants

- **I1** Demo mode never creates rows in Postgres/Cassandra for user content.
- **I2** Demo mode never uploads objects to SeaweedFS/S3 for user content.
- **I3** Env-forced Demo (`DEMO_MODE`) and client Demo both honor I1/I2 on the client path.
- **I4** Normal (non-Demo) path behaviour is unchanged.

## 8. Error model

| Code | When | User message |
|------|------|--------------|
| `demo_mode_readonly` | Server receives mutating request under DEMO_MODE | Demo deploy rejects server writes |
| `demo_quota` | IndexedDB/localStorage quota exceeded | Clear Demo data or free browser storage |

## 9. Observability

- Client: console info when Demo adapters handle a call (dev only).
- Server: warn log once at startup if `DEMO_MODE=true`; 403 responses include `code: demo_mode_readonly`.
- Health: may include `demoMode: true|false` when env set.

## 10. Security and privacy

- Demo data stays in the browser; do not send PII to api-server in Demo.
- Synthetic user must not be able to escalate to real OAuth sessions accidentally while Demo flag is set.
- Threat model: Demo deploy reduces data-at-rest risk on host; shared-browser risk increases (document Clear Demo Data).

## 11. Test plan

- Unit: demo store round-trip (projects CRUD); `isDemoMode` env + localStorage.
- Unit/integration: mutating API returns 403 when `DEMO_MODE=true`.
- Manual: `/demo` → create project/template → Network tab shows no mutating API.
- Lint + `tsc` in compose for front-cards and api-server.

## 12. Rollout and rollback

- Rollout: ship code; enable via `/demo` without env; or set `DEMO_MODE` on a dedicated host.
- Rollback: unset env flags; remove `/demo` link; prior persistence path unchanged.
- Feature flag keys in `.env.*.example`: `DEMO_MODE`, `NEXT_PUBLIC_DEMO_MODE` (default false).
