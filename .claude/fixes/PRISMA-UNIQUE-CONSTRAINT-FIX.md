# Prisma Unique Constraint Fix

**Date:** 2025-01-25
**Error:** `P2002: Unique constraint failed on the fields: (\`id\`)`
**Status:** ✅ FIXED

---

## Problem

**Error Message:**
```
Invalid `prisma.project.create()` invocation
Unique constraint failed on the fields: (`id`)

PostgreSQL: duplicate key value violates unique constraint "projects_pkey"
DETAIL: Key (id)=(default) already exists.
```

**Root Cause:**
- File: `api-server/src/core/prisma/client.ts:419`
- Code was hardcoding `id: 'default'` when creating projects
- The `id` field is the PRIMARY KEY (globally unique across ALL users)
- Multiple users trying to create default projects → same ID → constraint violation

**Schema Definition:**
```typescript
model Project {
  id        String   @id @default(uuid())  // Should auto-generate UUIDs
  userId    String
  name      String
  isDefault Boolean

  @@unique([userId, name])  // Unique project names PER USER
}
```

**What Was Wrong:**
```typescript
// ❌ WRONG - hardcoded global ID
project = await prisma.project.create({
  data: {
    id: 'default',  // Multiple users can't use same ID!
    userId,
    name: 'Default Project',
    isDefault: true
  }
});
```

---

## Solution Applied

### 1. ✅ Fixed Project Creation

**File:** `api-server/src/core/prisma/client.ts:409-452`

**Changes:**
- **Removed hardcoded `id: 'default'`** → Let Prisma auto-generate UUID
- **Added race condition handling** → Handle P2002 error if concurrent creates
- **Added retry logic** → Retry find if create fails due to concurrent request

**New Code:**
```typescript
async getOrCreateDefaultProject(userId: string) {
  try {
    // Try to find existing default project
    let project = await prisma.project.findFirst({
      where: { userId, isDefault: true }
    });

    if (!project) {
      // Create with auto-generated UUID (no hardcoded ID!)
      project = await prisma.project.create({
        data: {
          userId,
          name: 'Default Project',
          isDefault: true
          // id is auto-generated by @default(uuid())
        }
      });
    }

    return project;
  } catch (error: any) {
    // Handle race condition: another request created it
    if (error.code === 'P2002') {
      const project = await prisma.project.findFirst({
        where: { userId, isDefault: true }
      });
      if (project) return project;
    }
    throw error;
  }
}
```

**Result:**
- ✅ Each user gets their own default project with unique UUID
- ✅ No more global ID conflicts
- ✅ Handles concurrent requests safely

## 🔄 Update (2025-01-25) - Improved Race Condition Handling

### Additional Issue Found

**Error from `.misc\.errors`:**
```
ERROR:  duplicate key value violates unique constraint "projects_user_id_name_key"
DETAIL:  Key (user_id, name)=(invoketheoracle@gmail.com, Default Project) already exists.
```

**Cause:** The initial fix handled the `id` constraint but not the `@@unique([userId, name])` constraint. When concurrent requests for the same user tried to create projects simultaneously, both would pass the "check if exists" test and try to create, causing a constraint violation.

### Improved Solution

**Changed from check-then-create to atomic upsert:**

```typescript
// OLD (race condition window between check and create)
let project = await prisma.project.findFirst({ ... });
if (!project) {
  project = await prisma.project.create({ ... });  // Race here!
}

// NEW (atomic operation using unique constraint)
const project = await prisma.project.upsert({
  where: {
    userId_name: { userId, name: 'Default Project' }
  },
  update: { isDefault: true },
  create: { userId, name: 'Default Project', isDefault: true }
});
```

**Why This is Better:**
- ✅ **No race condition** - database ensures atomicity
- ✅ **Leverages unique constraint** - uses existing `@@unique([userId, name])`
- ✅ **Simpler** - one operation instead of find-then-create
- ✅ **Handles both errors** - P2002 on `id` AND on `(userId, name)`

### Scenarios Now Handled

1. ✅ **Multiple users creating projects** - Each gets unique UUID
2. ✅ **Same user, concurrent requests** - Upsert finds existing or creates atomically
3. ✅ **Multiple tabs/sessions** - No race condition
4. ✅ **Rapid page refreshes** - Idempotent operation

**Files Updated:**
- `api-server/src/core/prisma/client.ts` - Changed to use `upsert` instead of `findFirst` + `create`

**Status:** ✅ Complete - Both unique constraints now handled properly
